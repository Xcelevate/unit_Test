Mockito Verification: Basic verify()

Verify that mocked methods were called:

  • verify(mock).method() - Assert method was called once
  • verify(mock, times(n)).method() - Called exactly n times
  • verify(mock, never()).method() - Method was never called
  • verify(mock, atLeastOnce()).method() - Called at least once
  • verify(mock, atMostOnce()).method() - Called 0 or 1 time

Code Demo: verify()

@Test
void testVerifyMethodCalls() {
  UserRepository repo = 
    mock(UserRepository.class);
  UserService service = 
    new UserService(repo);
  
  service.createUser("John");
  verify(repo).save(any());
  
  service.findUserById(1);
  service.findUserById(2);
  verify(repo, times(2)).findById(any());
  
  service.deleteUser(3);
  verify(repo).delete(3);
}

Advanced Verification Modes

  • atLeast(n), atMost(n) - Flexible call count ranges
  • inOrder() - Verify call sequence across multiple mocks
  • timeout(millis) - Verify async method invocations
  • only() - Verify this is the ONLY call on mock
  • ignoreStubs() - Ignore stub-only interactions

Code Demo: InOrder Verification

@Test
void testCallSequenceWithInOrder() {
  Database db = mock(Database.class);
  Logger logger = mock(Logger.class);
  PaymentService service = 
    new PaymentService(db, logger);
  
  service.processPayment(100);
  
  InOrder order = inOrder(logger, db);
  order.verify(logger).log("Starting");
  order.verify(db).saveTransaction(any());
  order.verify(logger).log("Complete");
}

Code Demo: Timeout Verification

@Test
void testAsyncVerificationWithTimeout() {
  EventListener listener = 
    mock(EventListener.class);
  AsyncEventPublisher publisher = 
    new AsyncEventPublisher(listener);
  
  publisher.publishAsync("UserCreated");
  
  verify(listener, timeout(2000))
    .onEvent("UserCreated");
  
  verify(listener, timeout(1000).times(1))
    .onEvent(any());
}

Stubbing Void Methods

Special syntax for methods that return void:

  • doNothing().when(mock).voidMethod() - Default behavior
  • doThrow(exception).when(mock).voidMethod() - Simulate exception
  • doAnswer(answer).when(mock).voidMethod() - Custom logic
  • doCallRealMethod().when(mock).voidMethod() - Call real method
  • Use before stubbing for void methods (not when().thenReturn())

Code Demo: Void Stubbing

@Test
void testVoidMethodStubbing() {
  EmailService email = 
    mock(EmailService.class);
  
  doNothing().when(email)
    .sendNotification("john@example.com");
  
  doThrow(new MailException())
    .when(email).sendAlert("admin@ex.com");
  
  doAnswer(inv -> {
    String addr = inv.getArgument(0);
    System.out.println("Sending to: " + addr);
    return null;
  }).when(email).send(any());
  
  assertThrows(MailException.class,
    () -> email.sendAlert("admin@ex.com"));
}

ArgumentMatchers: Flexible Matching

  • any() - Matches any argument
  • eq(value) - Matches exact value
  • startsWith(), endsWith(), contains() - String matching
  • isA(Class.class) - Type matching
  • argThat(predicate) - Custom matching logic

Code Demo: ArgumentMatchers

@Test
void testArgumentMatchers() {
  PaymentGateway gateway = 
    mock(PaymentGateway.class);
  
  when(gateway.charge(any()))
    .thenReturn(true);
  when(gateway.charge(eq(0)))
    .thenThrow(new InvalidAmount());
  when(gateway.charge(
    argThat(amt -> amt > 10000)))
    .thenReturn(false);
  
  assertTrue(gateway.charge(100));
  assertFalse(gateway.charge(50000));
  assertThrows(InvalidAmount.class,
    () -> gateway.charge(0));
}

MockedStatic: Testing Static Methods

  • mockStatic(Class.class) - Mock all static methods
  • when(Static.method()).thenReturn() - Stub static method
  • verify(Static.method()) - Verify static call
  • Use try-with-resources for resource cleanup
  • Scope is limited to the resource block

Code Demo: MockedStatic

@Test
void testStaticMethodMocking() {
  try(MockedStatic uuid = 
      mockStatic(UUID.class)) {
    uuid.when(UUID::randomUUID)
      .thenReturn(UUID.fromString(
        "12345-12345-12345-12345"));
    
    String id = OrderService
      .generateOrderId();
    
    assertEquals(
      "12345-12345-12345-12345", id);
    uuid.verify(UUID::randomUUID);
  }
}

TDD Workflow: Red-Green-Refactor

  • RED - Write failing test (test doesn't compile)
  • GREEN - Write minimal code to pass test
  • REFACTOR - Improve code while keeping test green
  • Repeat for each feature or bug fix
  • Ensures code quality and 100% test coverage

Avoiding Over-Mocking

  • DON'T mock the class under test
  • DON'T mock simple value objects
  • DON'T mock just to verify method names
  • DO mock external dependencies (DB, API, cache)
  • DO test behavior, not implementation

Code Demo: Over-Mocking Antipattern

// BAD: Mocking class under test
@Test
void badOverMocking() {
  Calculator calc = mock(Calculator.class);
  when(calc.add(2, 3)).thenReturn(5);
}

// GOOD: Mock dependencies only
@Test
void goodMocking() {
  NumberFormatter formatter = 
    mock(NumberFormatter.class);
  Calculator calc = 
    new Calculator(formatter);
  
  int result = calc.add(2, 3);
  assertEquals(5, result);
}

Test Isolation: Key Principles

  • Each test is independent, no shared state
  • Tests can run in any order
  • @BeforeEach sets up fresh mocks
  • No test should depend on another test's result
  • Parallel test execution should work

Capstone: E-Commerce Order System

Test OrderProcessor with multiple dependencies:

  • OrderRepository (database mock)
  • PaymentGateway (external API mock)
  • InventoryService (business logic mock)
  • EmailService (notification mock)
  • Verify all interactions and edge cases

Capstone: Success Path Test

@ExtendWith(MockitoExtension.class)
class OrderProcessorTest {
  @Mock private OrderRepository repo;
  @Mock private PaymentGateway payment;
  @Mock private InventoryService inv;
  @Mock private EmailService email;
  @InjectMocks private OrderProcessor proc;

  @Test
  void shouldProcessOrderSuccessfully() {
    Order order = new Order(100, 5);
    when(inv.reserve(5)).thenReturn(true);
    when(payment.charge(100))
      .thenReturn("AUTHORIZED");
    
    String result = proc.process(order);
    
    assertEquals("SUCCESS", result);
    verify(repo).save(any());
    verify(email).sendConfirmation(any());
  }
}

Capstone: Failure Path Test

@Test
void shouldRejectWhenOutOfStock() {
  Order order = new Order(100, 100);
  when(inv.reserve(100))
    .thenReturn(false);
  
  assertThrows(OutOfStockException.class,
    () -> proc.process(order));
  
  verify(repo, never()).save(any());
  verify(payment, never()).charge(any());
  verify(email, never())
    .sendConfirmation(any());
}

@Test
void shouldRefundOnPaymentFailure() {
  when(inv.reserve(5)).thenReturn(true);
  when(payment.charge(any()))
    .thenThrow(new PaymentException());
  
  assertThrows(PaymentException.class,
    () -> proc.process(order));
  
  verify(inv).cancel(order.getId());
}

Common Pitfalls to Avoid

  • Over-mocking: Mocking dependencies and implementations
  • Brittle tests: Verifying implementation, not behavior
  • Missing exception paths: Only testing success cases
  • Ignoring async: Not testing concurrent execution
  • Poor test names: Unclear what test actually validates

Testing Async & Concurrent Code

  • @Timeout - Fail if test takes too long
  • verify(mock, timeout(millis)) - Async verification
  • CompletableFuture.join() - Block on async result
  • CountDownLatch - Synchronize threads in tests
  • ExecutorService - Test concurrent operations

Code Demo: Async Testing

@Test
@Timeout(5)
void testAsyncExecution() {
  EventListener listener = 
    mock(EventListener.class);
  AsyncExecutor exec = 
    new AsyncExecutor(listener);
  
  exec.executeAsync("task1");
  
  verify(listener, timeout(2000))
    .onComplete("task1");
}

@Test
void testConcurrentTasks() 
  throws InterruptedException {
  ExecutorService executor = 
    Executors.newFixedThreadPool(3);
  CountDownLatch latch = 
    new CountDownLatch(3);
  
  executor.submit(() -> {
    service.process();
    latch.countDown();
  });
  
  assertTrue(latch.await(5, 
    TimeUnit.SECONDS));
}

Edge Cases: Boundary Testing

  • Null inputs and empty collections
  • Boundary values (0, negative, max/min)
  • Unicode and special characters
  • Concurrent/race condition scenarios
  • Use @ParameterizedTest for multiple cases

Code Demo: Boundary Testing

@ParameterizedTest
@CsvSource({"0,0", "-5,Invalid", 
  "1000000,Valid"})
void testBoundaryValues(int val, 
    String expected) {
  if (val <= 0) {
    assertThrows(
      IllegalArgumentException.class,
      () -> service.process(val));
  } else {
    String result = 
      service.process(val);
    assertTrue(result.contains(expected));
  }
}

@Test
void testEmptyAndNull() {
  assertTrue(service.process(null)
    .isEmpty());
  assertTrue(service.process(
    new ArrayList<>()).isEmpty());
}

Testing Private Methods: When & How

  • DON'T test private methods directly (smell)
  • Private methods tested indirectly via public API
  • If private method is complex, extract to separate class
  • Refactor to public/protected if critical to test
  • Use Reflection sparingly (breaks encapsulation)

Real-World Testing Patterns

  • Test database interactions with mocked repositories
  • Mock external APIs and rate limiting
  • Verify email/notification sending
  • Test transaction rollback on failure
  • Validate state changes and side effects

Integration Testing (Beyond Unit Tests)

  • Unit Tests - Fast, isolated, mock all dependencies
  • Integration Tests - Real DB/APIs, slower but realistic
  • TestContainers - Docker for test databases
  • E2E Tests - Full system, staging environment
