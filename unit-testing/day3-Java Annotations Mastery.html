What Are Annotations?

Metadata that provides information about code without being part of code logic:

  • Do not directly affect code execution
  • Processed by compiler, tools, or runtime systems
  • Prefixed with @ symbol
  • Can target classes, methods, fields, constructors, parameters
  • Enable declarative programming and reduce boilerplate

Key Characteristics

  • Declarative - Define what, not how
  • Non-intrusive - Separate from business logic
  • Lightweight - No runtime overhead unless processed
  • Composable - Stack multiple annotations
  • Extensible - Create custom annotations easily

Common Built-in Annotations

  • @Override - Method overrides parent method
  • @Deprecated - Mark API as outdated
  • @SuppressWarnings - Suppress compiler warnings
  • @FunctionalInterface - Single abstract method interface
  • @SafeVarargs - Suppress varargs type warnings

Code Demo: @Override

public class Animal {
  public void sound() {
    System.out.println("Generic sound");
  }
}

public class Dog extends Animal {
  @Override
  public void sound() {
    System.out.println("Bark");
  }

  @Override
  public String toString() {
    return "Dog instance";
  }
}

Code Demo: @Deprecated

public class PaymentService {
  
  @Deprecated(since = "2.0",
    forRemoval = true)
  public void processPayment(int amount) {
    // Old implementation
  }
  
  @Deprecated(since = "1.5")
  public String getTransactionId() {
    return "TXN-001";
  }
  
  // New replacement method
  public PaymentResult charge(int amount) {
    return new PaymentResult(true);
  }
}

@SuppressWarnings: Common Cases

  • @SuppressWarnings("unchecked") - Raw type casting
  • @SuppressWarnings("deprecation") - Using deprecated API
  • @SuppressWarnings("all") - Suppress all warnings
  • Apply to method or class level
  • Use sparingly, fix root cause when possible

Code Demo: @FunctionalInterface

@FunctionalInterface
public interface Calculator {
  int calculate(int a, int b);
}

@FunctionalInterface
public interface StringProcessor {
  String process(String input);
}

// Usage with Lambda

Calculator multiply = (a, b) -> a * b;

Creating Custom Annotations

  • Use @interface keyword to declare
  • Define annotation elements (methods)
  • Specify default values for optional elements
  • Apply meta-annotations for behavior
  • Use reflection to process at runtime

Code Demo: Custom Annotation

@Target({ElementType.METHOD, 
  ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface ValidateUser {
  String role() default "USER";
  int maxRetries() default 3;
  long timeout() default 5000L;
}

@ValidateUser(role = "ADMIN", 
  maxRetries = 5)
public void deleteUser(int id) {
  System.out.println("Admin operation");
}

@ValidateUser
public void viewProfile(int id) {
  System.out.println("User operation");
}

Meta-annotation: @Target

Specifies where annotation can be applied:

  • ElementType.TYPE - Classes, interfaces, enums
  • ElementType.METHOD - Method declarations
  • ElementType.FIELD - Field variables
  • ElementType.PARAMETER - Method parameters
  • ElementType.CONSTRUCTOR - Constructor declarations

Code Demo: @Target Variations

// Type-level annotation
@Target(ElementType.TYPE)
public @interface Service { }

// Method-level annotation
@Target(ElementType.METHOD)
public @interface CacheResult { }

// Field-level annotation
@Target(ElementType.FIELD)
public @interface Inject { }

// Multiple targets
@Target({ElementType.METHOD, 
  ElementType.TYPE})
public @interface Secure { }

@Service
class UserService {
  @Inject
  private UserRepository repo;
  
  @CacheResult
  public User findById(int id) {
    return repo.get(id);
  }
}

Reflection: Processing Annotations

  • isAnnotationPresent() - Check if annotation exists
  • getAnnotation() - Get single annotation instance
  • getAnnotations() - Get all annotations
  • getDeclaredAnnotations() - Get direct annotations only
  • Works on classes, methods, fields, constructors, parameters

Code Demo: Field Annotations

public @interface Validate {
  String pattern() default "";
  int minLength() default 0;
  int maxLength() default 255;
  boolean required() default true;
}

public class User {
  @Validate(minLength = 3, 
    maxLength = 50)
  private String username;
  
  @Validate(pattern = "^[A-Za-z0-9+_.-]+@(.+)$")
  private String email;
  
  @Validate(minLength = 8)
  private String password;
  
  private String address; 
    // No validation
}

// Process fields
for (Field f : User.class
    .getDeclaredFields()) {
  if (f.isAnnotationPresent(Validate.class)) {
    Validate v = f.getAnnotation(
      Validate.class);
  }
}

Annotation Processing (Compile-Time)

  • Process annotations during compilation
  • Generate new source/class files automatically
  • Extend AbstractProcessor class
  • Use in build pipeline with Maven/Gradle
  • Examples: Lombok, MapStruct, Dagger

Code Demo: Annotation Processor

@SupportedAnnotationTypes(
  {"com.example.GenerateBuilder"})
@SupportedSourceVersion(
  SourceVersion.RELEASE_21)
public class BuilderProcessor 
  extends AbstractProcessor {
  
  @Override
  public boolean process(
    Set 
      annotations,
    RoundEnvironment roundEnv) {
    
    for (Element elem : roundEnv
        .getElementsAnnotatedWith(
          GenerateBuilder.class)) {
      
      // Generate builder code
      generateBuilder((TypeElement) elem);
    }
    return true;
  }
}

Code Demo: Validation Annotation

@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Email {
  String message() 
    default "Invalid email";
}

public class User {
  private String name;
  
  @Email
  private String email;
}

public class Validator {
  public static  void validate(T obj) 
    throws ValidationException {
    Class clazz = obj.getClass();
    
    for (Field field : 
      clazz.getDeclaredFields()) {
      if (field.isAnnotationPresent(
        Email.class)) {
        field.setAccessible(true);
        String value = 
          (String) field.get(obj);
        
        if (!isValidEmail(value)) {
          throw new ValidationException(
            "Invalid email");
        }
      }
    }
  }
}

Best Practices for Annotations

  • Always specify @Target and @Retention
  • Use @Documented for public-facing annotations
  • Provide meaningful default values
  • Document annotation purpose and usage
  • Avoid overuse - not every feature needs one

When to Avoid Annotations

  • Frequently changing requirements
  • Business logic that should be in code
  • When explicit code is clearer
  • Simple one-off markers (use comments)
  • Complex configuration (use XML or JSON)
